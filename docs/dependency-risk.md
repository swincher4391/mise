# External Dependency Risk

Closes finding **E3** of `docs/site-improvement-audit-2026-07.md` (P2). Widens PRD §3.6 (which covers only the HuggingFace SPOF) into a full per-dependency risk table for the dependencies added since the PRD, plus a closing recommendation for a lightweight weekly uptime smoke-check.

Failure modes below are grounded in the actual integration code where possible; inferences are marked *(inferred)*.

---

## Risk table

| Dependency | What it powers | Failure mode | Current fallback | Recommended mitigation |
| --- | --- | --- | --- | --- |
| **HuggingFace router** (`router.huggingface.co`) | LLM + vision inference for nearly everything AI: photo import & video-frame OCR (Qwen2.5-VL, `extract-image.ts`, `proxy-browser.ts`), Whisper transcription, Describe/chat + ingredient normalization (`recipe-chat.ts`), and caption structuring (`yt-transcript.ts`). | **Single point of failure.** If HF tightens the free tier, changes pricing, or has an outage, *all* AI features degrade at once. Compounding risk: models must be **provider-pinned** (`:hyperbolic`, `:featherless-ai`) — un-pinned or deprovisioned models fail with `model_not_supported` (five model-churn commits in the git log confirm this is a live, recurring break). | **Partial.** Photo import has a Tesseract.js OCR client-side fallback (lower quality). Describe/chat/normalize models are read from env vars (`STRUCTURE_MODEL`, `CHAT_MODEL`, `NORMALIZE_MODEL`) so a pinned model can be swapped without a redeploy. Video extraction and vision OCR have **no** fallback. | Keep the env-var model indirection for *all* HF calls (vision URLs are currently hardcoded — move them to env too). The router exposes an OpenAI-compatible API, so a secondary provider (Groq/Together/Fireworks) is a low-code failover — document the swap procedure. Monitor free-tier usage/cost. Treat model deprovisioning as an expected event, not an incident. |
| **Supadata** (`api.supadata.ai`) | YouTube caption retrieval (`yt-transcript.ts`) — the server-side relay that fetches captions YouTube blocks from datacenter IPs, before LLM structuring. | **Hard monthly cap: 100 credits/month** on the free tier (one GET per YouTube extract). Once exhausted, server-side YouTube extraction returns "No captions available" for the rest of the month. Also a SPOF for the server YouTube path — if Supadata is down or changes its API, that path breaks. | The endpoint already returns a clean "No captions available" error rather than hanging. The **Chrome extension** is the real hedge: it fetches captions client-side (no datacenter block, no Supadata credit), bypassing this dependency entirely. | Track credit burn (log/alert at ~80/month). Treat the extension as the primary YouTube path and Supadata as best-effort. Evaluate paid Supadata tier only if server-side YouTube volume justifies it — otherwise let the extension carry it (per finding A2/E2, YouTube is already extension-first). |
| **tmpfiles.org** | Temporary image hosting so the HF vision model can fetch user images by URL: cookbook-photo import (`extract-image.ts`) and video keyframe grids (`_lib/frameExtraction.ts`). | **Uptime + privacy optics.** It's a free, best-effort file host with no SLA — if it's down or slow, photo import and frame-OCR fail. *(inferred)* Privacy angle: user-uploaded cookbook photos / video frames transit a third-party public file host, which sits awkwardly next to the "everything is local" positioning even though uploads auto-expire and aren't persisted by Mise. | Uploads auto-expire (temporary hosting). No alternate host wired in — if tmpfiles is down, these paths fail. Tesseract.js OCR partially covers photo import (text-only, no vision). | Add a fallback host or, better, pass images to the vision model as **base64 data URLs** where the model/provider supports it, removing the third-party hop entirely. If tmpfiles stays, disclose it on the privacy page (pairs with B2) and confirm the CSP `connect-src` entry stays scoped. Consider a self-hosted short-TTL upload via Vercel Blob. |
| **DuckDuckGo HTML scraping** (`html.duckduckgo.com/html`, `lite.duckduckgo.com` fallback) | Discover feature web search (`recipe-discover.ts`) — scrapes DDG HTML SERP, then enriches results with JSON-LD preview cards. | **Silent breakage.** This parses DDG's HTML markup (`parseDdgResults`). DDG can change markup, rate-limit, or block the datacenter IP at any time with no notice — the parser then returns zero results and Discover quietly shows nothing. No contract, no API, no warning. | A secondary `lite.duckduckgo.com` table-format parser exists as a fallback. Blocked-domain filtering (`isBlockedDomain`, e.g. allrecipes.com) runs on results. Product-level, "try Describe" is offered as an alternative path. | Add a **result-count assertion**: if both DDG parsers return zero for a known-good query, surface a distinct "search unavailable" state (not an empty list) and log it — so silent breakage becomes visible. Long-term, evaluate a proper search API (Brave/Bing/SerpAPI) as a paid, contractual replacement if Discover proves load-bearing. |
| **Vercel KV** (`@vercel/kv`) | Per-IP rate limiting + PIN lockout (`_lib/rateLimit.ts`) and the share-URL sitemap store (`sitemap-xml.ts`). | **Silent security downgrade.** If `KV_REST_API_URL`/`KV_REST_API_TOKEN` are unset or KV is unreachable, the limiter falls back to **per-instance in-memory counters** — an attacker gets a fresh bucket per warm lambda and PIN lockout stops being shared. The code itself flags this as "the only thing standing between a scraper and the API bill." *(The KV-downgrade commit noted in the audit suggests this already wobbled once.)* Sitemap loses its backing store → share-URL sitemap empties. | The limiter **degrades rather than fails** (still limits per-instance) and logs loudly once (`console.error` on missing config / unreachable KV). Not silent in logs — but silent to users. | Add a startup/health assertion that KV is configured in production and alert if the "falling back to per-instance" warning ever fires. Given rate limiting is the real cost control for the unauthenticated AI endpoints (finding E4), treat KV availability as production-critical, not best-effort. |
| **Stripe** (`api/create-checkout.ts`, `api/verify-purchase.ts`) | Payments — the $4.99 unlock: Checkout session creation and purchase verification/restore. | *(inferred)* Stripe outage or API/key issue blocks new purchases and restores. Note (finding B3): these two endpoints are currently **unthrottled** — `create-checkout` can be scripted to mass-create sessions (cost/fraud-signal on the Stripe account) and `verify-purchase` is an unthrottled purchase-enumeration oracle. So the risk here is as much abuse as uptime. | Stripe is highly reliable; core app works offline/local without payment (paywall is a filter, not the core). No throttling fallback today. | Apply the shared per-IP limiter to both endpoints (already the top item in finding B3). Use Stripe webhooks/idempotency keys defensively. Purchase state is cached client-side, so a transient Stripe outage doesn't lock out already-unlocked users. |
| **Instacart / Impact** (`connect.instacart.com` via `api/grocery/instacart.ts`; Impact affiliate params) | "Get Ingredients" deep links (recipe-level and shopping-list-level) and the **affiliate revenue** attached to them (Impact `utm_content=...partnerid-<IMPACT_PARTNER_ID>`). | *(inferred)* Instacart API change/outage breaks the "Get Ingredients" button; a missing/misconfigured `INSTACART_API_KEY` returns a 500; a broken/removed `IMPACT_PARTNER_ID` silently drops affiliate attribution (links still work, **revenue tracking is lost with no error**). Partner-agreement/compliance changes could also gate the integration. | Endpoint is rate-limited (`name: 'instacart'`) and returns a user-friendly error on API failure. The grocery list itself works without Instacart (manual/export). Affiliate params are appended defensively (URL still valid if partner id absent). | Add a check that `IMPACT_PARTNER_ID` is present in production (lost attribution = lost revenue, invisibly). Monitor Instacart API responses for schema drift. Keep affiliate disclosure current (finding B1) to stay compliant with Impact/Instacart partner terms. |
| **Publisher IP-blocking** (People Inc. / large recipe publishers) | URL extraction via the proxy (`api/proxy.ts`, `api/proxy-browser.ts`) — the JSON-LD/heuristic extractor fetching publisher recipe pages from Vercel's datacenter IPs. | **Targeted counter-measure — already happening.** People Inc. sites now **hard-block the extractor's datacenter IP** (403 / block page). This is the first sign of *deliberate* anti-extraction defense, not incidental bot filtering — and the expectation is that **more publishers follow** (Allrecipes/Food Network-class sites are the likely next wave). Server-side extraction of these domains degrades to failure. | Post-redirect URL validation and a blocked-domains list exist. Product offers other input modes (Paste, Photo, Describe). The **Chrome extension** is again the strategic hedge — it extracts from the *user's own browser session/IP*, sidestepping datacenter blocks entirely. | Lean into the extension as the durable extraction path for hostile publishers (aligns with A2). Detect block pages (403 / challenge markup) and return a clear "this site blocks automated extraction — try the extension or Paste" message instead of a generic failure. Track which publishers block, as a leading indicator of where the extension becomes mandatory. Do **not** invest in IP-rotation/evasion — it's an arms race against the exact partners a legit product wants to stay clean with. |

## Closing recommendation: weekly uptime smoke-check

Every dependency above fails in a way that is **invisible to the team until a user reports it** — DDG markup drift returns empty results, Supadata silently hits its cap, tmpfiles goes down, a HF model gets deprovisioned, KV falls back per-instance, a publisher starts blocking. None throws an alert today.

Add a **lightweight weekly smoke-check** (manual to start, then a scheduled job / CI cron) that exercises the three critical pipelines against **known-good fixtures** and asserts non-empty, well-formed output:

- **Extract** — a stable JSON-LD recipe URL (one known-good publisher) → expect title + ingredients + steps.
- **Discover** — a fixed query (e.g. "chicken tikka masala") → expect ≥1 result card (catches DDG markup breakage).
- **Video** — a known YouTube URL with captions → expect a structured recipe (catches Supadata cap/HF/model-pin breakage).

Optionally add a photo-import fixture (catches tmpfiles + vision-model breakage) and a KV-configured assertion. Keep it thin: the point is a **weekly signal that the external surface still works**, turning today's silent failures into a dated, visible check. This dovetails with the E1 Playwright smoke suite — the same specs can run against known-good external URLs on a weekly schedule rather than per-PR.
